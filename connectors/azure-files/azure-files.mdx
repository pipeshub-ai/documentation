## Overview

Azure File Shares provide **SMB-compatible** cloud file storage with a **real directory hierarchy**. Files and folders are organized under storage accounts and file shares, making them ideal for documents, team shares, and application data that expect a traditional file system layout.

The Azure File Shares connector enables you to sync file shares, directories, and files from your Azure Storage account into PipesHub, making your content searchable through AI-powered search and easily accessible across your organization.

### Azure File Shares Data Structure

The connector understands Azure File Storage's hierarchy:

**Storage Account → File Shares → Directories → Files**

| Entity | Description |
|--------|-------------|
| **Storage Account** | Top-level Azure resource that owns one or more file shares |
| **File Shares** | Named shares (like network drives) that contain directories and files |
| **Directories** | Real folders with metadata; can be nested recursively |
| **Files** | Individual files stored in directories with paths and metadata |

### What Gets Synced

The connector indexes the following content for AI-powered search:

- **Files**: All supported file types stored in selected shares (documents, images, archives, etc.)
- **Directories**: Real folder structure with metadata
- **Metadata**: File names, sizes, modification dates, MIME types
- **Permissions**: Access scope (Personal vs Team) controlled via connector configuration

### Setup Overview

The Azure File Shares connector uses an **Azure Storage connection string** from the Azure Portal. It syncs file shares, directories, and files, enabling comprehensive search and access across your Azure content.

### Authentication

The connector uses **Connection String authentication**:

- Connection string is generated from your Azure Storage account
- Contains account name, key, and endpoint
- Stored securely in PipesHub as a secret

<Info>
  Connection strings provide full access to your storage account. Treat them like passwords and store them securely. Rotate keys regularly following Azure security best practices.
</Info>

### How to configure and enable the Azure File Shares Connector

### Step 1: Get the Azure Storage Connection String

1. **Sign in to Azure Portal:**  
   Go to `https://portal.azure.com` and sign in with your Azure account.

2. **Open your Storage Account:**
   - In the left navigation, select **"Storage accounts"**
   - Click the storage account you want PipesHub to use

3. **Navigate to Access Keys:**
   - In the storage account menu, go to **"Security + networking" → "Access keys"** (or **"Access keys"** section)

### Step 4: Configure Filters (Optional)

Filters control which data is **synced** and which data is **indexed** for search.

#### Sync Filters

Sync filters decide what content is downloaded from Azure File Storage. Data excluded by sync filters is never fetched.

**Available Sync Filters:**

1. **File Share Names**
   - **Operator**: `In` (include only) or `Not In` (exclude)
   - **Selection**: Choose from a **searchable dropdown** of file shares in your storage account
   - Select one or more shares to scope the sync

2. **File Extensions**
   - **Operator**: `In` (include) or `Not In` (exclude)
   - **Value**: Comma-separated list of extensions, e.g. `pdf, docx, txt`
   - Case-insensitive; do not include leading dots

3. **Modified Date**
   - **Operators**: `Is After`, `Is Before`, `Is Between`
   - Use to sync only recently updated files (e.g., last 6 months)

4. **Created Date**
   - **Operators**: `Is After`, `Is Before`, `Is Between`
   - Use to include or exclude older files based on creation time

#### Indexing Filters

Indexing filters control what synced data is processed for AI-powered search.

- **Index Files** (default: enabled) – Include file content in search
- **Index Folders** (default: enabled) – Include directory structure in search

**Example Configurations:**

- **Sync only certain shares**: File Share Names → `In` → select shares
- **Documents only**: File Extensions → `In` → `pdf, docx, txt, pptx`
- **Exclude media**: File Extensions → `Not In` → `jpg, png, mp4, mov`
- **Recent files only**: Modified Date → `Is After` → `YYYY-MM-DD`

<Info>
  Filters can be changed anytime. Updates take effect on the next sync.
</Info>

### Supported Features

The Azure File Shares connector syncs:

- **File Shares** as top-level record groups
- **Files** with metadata such as name, size, last modified time, MIME type
- **Directories** (real folders) with full hierarchy and metadata
- **Permissions** via connector scope (Personal vs Team)

### Data Sync Behavior

#### Initial Sync

The first sync scans the selected file shares:

- Discovers file shares and creates corresponding record groups in PipesHub
- Recursively traverses directories and lists files within each share
- Extracts metadata and builds the directory hierarchy
- Indexes supported file content for AI-powered search

<Info>
  Use file share and file extension filters to significantly reduce initial sync time, especially for large storage accounts.
</Info>

#### Incremental Sync

Subsequent syncs are incremental:

- Uses file timestamps and sync points per share (`last_sync_time`) to fetch only new or updated content
- Tracks sync state per share to avoid reprocessing unchanged data
- Handles deletions by marking records as deleted and removing them from search results

### Permission Handling

The connector respects Azure security and PipesHub scopes:

- **Azure-level permissions**: Connector can only access shares and files allowed by the connection string
- **PipesHub scopes**:
  - **Personal**: Only the connector creator can see synced content
  - **Team**: All organization users have read access to synced content
- **Permission inheritance**: Files inherit access from their share and connector scope

### Sync Frequency

**Scheduled Sync:**
- Default: ~60 minutes
- Options: 15 min, 30 min, 1 hour, 4 hours, 24 hours

**Manual Sync:**
- Triggered from the connector UI
- Recommended for testing or after large content changes

<Info>
  For very large file shares or infrequently updated content, prefer less frequent syncs (4–24 hours) to optimize cost and performance.
</Info>

### Azure File Storage Compatibility

This connector is designed specifically for **Azure File Storage** and uses Azure's official SDKs and APIs.

It supports:

- **All Azure regions** supported by Azure Files
- **Standard redundancy options** (LRS, ZRS, GRS, RA-GRS etc.)
- **SMB and NFS** protocol endpoints; connector uses REST API for sync

<Info>
  The connector is optimized for file share content used in knowledge management and document search scenarios.
</Info>

## Troubleshooting

<AccordionGroup type="single" collapsible>

<Accordion title="Troubleshooting" icon="circle-exclamation">

**Invalid connection string:**

*Symptoms:*
- Error messages like "Authentication failed" or "Invalid connection string"
- Connector fails to initialize

*Solutions:*
1. Verify the connection string is copied completely from Azure Portal
2. Ensure there are no extra spaces or line breaks
3. Regenerate and replace the access keys if they may have been rotated
4. Confirm that the storage account is active and not deleted

**Access denied or missing shares:**

*Symptoms:*
- Some file shares do not appear in the share filter
- Sync fails with permission-related errors

*Solutions:*
1. Confirm you are using a connection string with correct permissions
2. Check share-level access policies in Azure Portal
3. Ensure the storage account is in the correct subscription and region

**No data syncing:**

*Symptoms:*
- Connector shows "Active" but no files or folders appear
- Indexing progress remains at 0%

*Solutions:*
1. Check that at least one file share is selected in filters
2. Verify file extension filters are not excluding all relevant file types
3. Confirm date filters are not set too restrictively
4. Validate that files actually exist in the selected shares

**Sync taking too long:**

*Symptoms:*
- Initial sync runs for many hours
- Progress appears slow for very large shares

*Solutions:*
1. Use file share filters to limit scope
2. Restrict file extensions to document-heavy types (e.g., `pdf, docx, txt`)
3. Use date filters to focus on recent content
4. Increase sync interval for large storage accounts

**Files not appearing in search:**

*Symptoms:*
- Files are visible in connector records, but not searchable

*Solutions:*
1. Ensure **Index Files** is enabled in indexing filters
2. Confirm the file type is text-based and supported for indexing
3. Check that files are not encrypted or password-protected
4. Allow time for indexing to complete, especially for large files

## FAQ

<AccordionGroup type="single" collapsible>

<Accordion title="How long does the initial sync take?">
  Sync time depends on the number and size of files and directories:
  
  | Storage Size | Estimated Time |
  |--------------|----------------|
  | Small (< 1,000 files) | 5–15 minutes |
  | Medium (1,000–10,000 files) | 15–60 minutes |
  | Large (10,000–100,000 files) | 1–3 hours |
  | Very Large (100,000–1,000,000 files) | 3–8+ hours |
  
  Use file share and extension filters to reduce initial sync time dramatically.
</Accordion>

<Accordion title="Can I connect multiple storage accounts?">
  **Yes.** Create one Azure File Shares connector instance per storage account.  
  Each instance uses its own connection string and filter configuration.
</Accordion>

<Accordion title="What file types can be indexed for search?">
  Text-based and document formats are indexed, including:
  
  - Documents: PDF, DOCX, PPTX, XLSX, ODT, RTF  
  - Text: TXT, MD, CSV, JSON, XML, HTML  
  - Code files (e.g., `.py`, `.js`, `.ts`, `.java`)
  
  Binary-only formats (images, videos, archives) are synced and visible but their contents are not searchable.
</Accordion>

<Accordion title="What happens if I delete a file in Azure?">
  Deletions are detected during the next incremental sync:
  
  - The record is marked as deleted in PipesHub  
  - The file is removed from search results  
  - Historical behavior depends on your organization's retention policies
</Accordion>
